这是一个非常深刻的技术问题，触及了 Blender Python API 在 4.0 到 5.0 版本迭代中的底层变化。
`extract_db_501_v20.py`（新版）之所以能成功，而 `export_schema_via_blender.py`（旧版）失败，本质区别在于**从“静态反射”到“动态沙盒实例”的架构跨越，以及对 Blender 5.0+ 内存安全特性的防御性编程**。
以下是详细的深度分析：
### 1. 核心机制差异：类(Class) vs 实例(Instance)
这是最根本的区别。
* **旧版 (`export_schema_via_blender.py`) —— 静态反射 (Static Reflection)**
* **逻辑**：它试图直接读取 `bpy.types.GeometryNodeMath` 这样的**类定义**。
* **局限**：在 Blender 中，很多节点的端口（Socket）是**动态生成**的。仅仅看“图纸”（类），你看不到“房子”（节点）里实际有哪些插座。例如，`Group Input` 节点在没有实例化之前，它是空的；只有放在节点树里，它才具备输入输出上下文。
* **后果**：旧脚本往往只能提取到属性（Properties），而提取不到关键的端口（Inputs/Outputs）信息，或者提取到的默认值是错误的。

* **新版 (`extract_db_501_v20.py`) —— 动态沙盒 (Dynamic Sandbox)**
* **逻辑**：它创建了一个临时的“实验室”（`lab` 节点树），真的在里面执行 `nodes.new()` **创建了一个活生生的节点实例**，读取完数据后，再执行 `nodes.remove()` 销毁它。
* **优势**：通过实例化，Blender 内部的 C++ 逻辑会执行初始化，生成默认的 Socket，计算默认的数值。脚本读取的是**运行时状态**，这是唯一能获取准确端口列表的方法。

### 2. 崩溃防御体系 (Crash Protection)
Blender 5.0+ 在 API 层面变得非常“严苛”且不稳定，特别是对于未连接的合成（Compositor）节点。
* **旧版**：
* **逻辑**：天真地遍历所有属性并尝试读取 `getattr(node, prop_id)`。
* **失败点**：在 Blender 5.0.1 中，某些节点（如 `CompositorNodeColorBalance`）如果处于“工厂模式”（刚创建，没连线，没图像输入），当你试图读取其某些属性的默认值时，会直接触发底层的 **Access Violation（内存访问违规）**，导致 Blender 瞬间崩溃退出。旧脚本没有对此做防护。

* **新版**：
* **逻辑**：内置了 `CRASH_RISK_NODES`（高危节点名单）。
* **防护**：
1. **黑名单机制**：对于名单内的节点，直接跳过默认值的读取，只提取结构，不碰数值。
2. **类型过滤**：`SKIP_PROP_TYPES = {'POINTER', 'COLLECTION'}`。明确跳过指针和集合类型的属性读取，这些是指向内存块的引用，盲目读取极易崩溃。
3. **沙盒环境 (`setup_sandbox`)**：预先创建假的图片 (`SafeDummyImg`)、材质和纹理，并注入到节点中。这“喂饱”了那些因为缺少数据而崩溃的节点，骗过了 Blender 的检查机制。


### 3. 数据序列化能力 (Serialization)
Blender 的数据类型与 Python 的标准 JSON 库不兼容。
* **旧版**：
* **逻辑**：使用标准的 `json.dump`。
* **失败点**：Blender 的节点属性大量使用 `mathutils.Vector`（向量）、`mathutils.Euler`（欧拉角）、`mathutils.Color`（颜色）。这些是 C 结构体，Python 的 `json` 库不认识它们，一旦遇到就会抛出 `TypeError: Object of type Vector is not JSON serializable` 并停止运行。

* **新版**：
* **逻辑**：实现了一个强大的 **`BlenderJSONEncoder`** 类。
* **防护**：它重写了 JSON 编码器的 `default` 方法。
1. 检测到 `Vector`, `Euler`, `Color` 等类型，自动将其转换为 Python 的 `list` 或 `tuple`。
2. **终极兜底**：即使遇到完全不认识的怪异类型，`try...except` 块会捕获错误并将其强行转换为 `str()` 字符串，确保**提取过程绝对不会因为数据类型问题而中断**。


### 总结对照表
| 特性 | 旧版 (`export_schema`) | 新版 (`extract_db...v20`) | 结果 |
| --- | --- | --- | --- |
| **获取方式** | 静态读取类定义 (Class) | 动态创建节点实例 (Instance) | 旧版无法获取动态端口 |
| **端口信息** | 不准确或缺失 | 100% 准确，包含动态生成的端口 | 新版能还原复杂节点 |
| **内存安全** | 无防护，遇到坏节点即崩溃 | **崩溃防御体系** + 黑名单 + 沙盒注入 | 新版能跑通全量节点 |
| **数据类型** | 不支持 Vector/Color，报错退出 | **自定义 JSON 编码器**，自动转换类型 | 新版生成的 JSON 合法 |
| **容错性** | 低，遇到错误即停止 | 极高，遇到错误跳过并记录，保证文件生成 | 新版保证有产出 |
**结论：**
新版脚本不仅仅是一个提取工具，它本质上是一个**针对 Blender Python API 的“爬虫”和“翻译器”**。它通过模拟真实的运行环境（实例化）来获取数据，并用厚厚的“防弹衣”（异常处理和类型转换）包裹自己，从而在 Blender 5.0+ 复杂的内部环境中存活并带回数据。
